# 천재적 사고 엔진 시스템 (Genius Thinking Engine System)

서브에이전트 위임 아키텍처(STSA)로 동작한다. 메인 에이전트는 Cynefin 진단 + 모듈 선택(~300tok)만 수행하고, 전체 분석은 서브에이전트에 위임하여 메인 컨텍스트를 95% 절감한다.

## 사전 단계: 경로 해석

이 스킬의 모든 파일 참조는 설치 위치에 따라 동적으로 해석된다:

1. `@references/thinker-prompt.md`를 Read 도구로 읽는다 (내용은 2단계에서 사용)
2. 읽힌 파일의 절대경로에서 파일명(`thinker-prompt.md`)과 디렉토리명(`references/`)을 제거하여 **스킬 루트 경로**를 구한다
   - 예: `/home/user/smartthink/skill/references/thinker-prompt.md` → 스킬 루트 = `/home/user/smartthink/skill`
   - Read 도구가 절대경로를 반환하지 않으면: `Glob("**/smartthink/skill/references/thinker-prompt.md")`로 위치를 탐색한다
3. 변수를 설정한다:
   - `REFERENCES_DIR` = `{스킬 루트}/references`
   - `DATA_DIR` = `{스킬 루트}/.data`
4. `DATA_DIR` 존재 확인: 디렉토리가 없으면 `mkdir -p {DATA_DIR}/briefs`로 `.data/` 및 `.data/briefs/` 모두 생성한다

## 모드 라우팅

`$ARGUMENTS`가 비어있거나 공백만 있으면 → 사용법을 안내하고 종료한다: "`/smartthink <주제>` 형식으로 입력하세요."

`$ARGUMENTS`를 확인하여 모드를 먼저 판별하라:

| 조건 | 모드 | 동작 |
|------|------|------|
| `$ARGUMENTS`가 "light " 또는 "경량 "으로 시작 | **Light** | 주제 = 접두사 이후 텍스트 → "Light 모드 실행" 섹션 |
| `$ARGUMENTS`가 "search " 또는 "검색 "으로 시작 | **Deep + Search** | 주제 = 접두사 이후 텍스트 → "Deep 모드 실행" (검색 데이터 포함) |
| 그 외 | **Deep** | 주제 = $ARGUMENTS 전체 → "Deep 모드 실행" |

```
IF $ARGUMENTS가 "light " 또는 "경량 "으로 시작하면
  → 주제 = $ARGUMENTS에서 접두사 이후의 텍스트
  → "Light 모드 실행" 섹션으로 이동
ELSE IF $ARGUMENTS가 "search " 또는 "검색 "으로 시작하면
  → 주제 = $ARGUMENTS에서 접두사 이후의 텍스트
  → SEARCH_MODE = "search 모드 — WebSearch 사용"
  → WebSearch 호출: WebSearch("<주제>") (도구 미사용 환경이면 폴백으로 직행)
    - 폴백: 실패 또는 도구 미사용 시 SEARCH_DATA = "검색 실패", SEARCH_MODE = "검색 실패 — 내부 모듈만 사용"
    - 성공 시 SEARCH_DATA = 핵심 3~5개 포인트로 압축한 검색 결과 (총 500자 이내)
  → "Deep 모드 실행" 섹션으로 이동
ELSE
  → 주제 = $ARGUMENTS 전체
  → SEARCH_DATA = "없음"
  → SEARCH_MODE = "내부 모듈만 사용"
  → "Deep 모드 실행" 섹션으로 이동
```

## Deep 모드 실행

Deep 모드는 서브에이전트에게 전체 분석을 위임한다. 메인 에이전트는 진단+모듈 선택만 수행한다.

### -1단계: 진화 상태 로딩 (Evolution State Loading)

- `{DATA_DIR}/evolution-state.md` 파일을 읽는다 (사전 단계에서 해석한 경로 사용)
- 파일이 존재하지 않거나 인사이트가 없으면 (플레이스홀더 텍스트만 있으면) → EVOLUTION_STATE = "없음"으로 설정하고 0단계로 진행
- `## 핵심 인사이트` 헤더가 없으면 → 손상된 상태로 간주하고 경고 출력 후 EVOLUTION_STATE = "없음"으로 설정하고 0단계로 진행 (서브에이전트 Step 6에서 초기 템플릿을 자동 생성한다)
- 파일에 인사이트가 있으면 (`(0/10)`이 아닌 실제 인사이트 슬롯이 존재) → EVOLUTION_STATE 변수에 파일 전체 내용을 저장
- **편향 주의**: 과거 인사이트가 현재 문제에 부적합할 수 있다. 0단계 Cynefin 진단 결과가 인사이트보다 항상 우선한다.

### 0단계: 상황 진단 (Cynefin)

- 주제를 파악하고 **Cynefin 프레임워크**로 문제 도메인을 분류하라

| 도메인 | 특성 | 라우팅 |
|--------|------|--------|
| **명확 (Clear)** | 인과관계가 명백 | SmartThink 스킵 |
| **복잡 (Complicated)** | 분석하면 답이 나옴 | 권장: 핵심 엔진 + 인지 무기고. TRIZ는 기술적 모순이 핵심일 때만 |
| **복합 (Complex)** | 창발적 | 권장: 패턴 합성 + 현실 왜곡. 보조: TRIZ, 안티프래질 |
| **혼돈 (Chaotic)** | 긴급 대응 | 권장: 실행 속도. 분석 과다 주의 |
| **무질서 (Disorder)** | 불명확 | 제1원리 분해 → 재진단 |

- **명확(Clear)** 도메인이면 SmartThink 전체 프로세스를 건너뛰고 간단히 답변하라

### 1단계: 주제 분류 및 모듈 선택

- 아래 결정 트리로 범위를 좁힌 뒤, 유형 테이블에서 최종 분류하라:

#### 분류 결정 트리
> Q1. 외부 시장/고객이 핵심인가? → Y: Q2a / N: Q2b
> Q2a. 새로 만드는 것인가? → Y: 아이디어 발굴 or 사업 구축 / N: 전략 수립 or 기회 분석
> Q2b. 기술적 모순(물리/공학적 트레이드오프)이 핵심인가? → Y: 체계적 발명 / N: Q3
> Q3. 선택지 간 비교 판단인가? → Y: 의사 결정 / N: 문제 해결

- 결정 트리 결과를 참고하여 아래 테이블에서 최종 유형과 모듈을 선택하라:

| 유형 | 설명 | 우선 참조 모듈 |
|------|------|--------------|
| 아이디어 발굴 | 새로운 비즈니스/제품 아이디어 탐색 | 핵심 엔진 + 현실 왜곡 + 패턴 합성 |
| 사업 구축 | 0→1 비즈니스 빌딩 | 유니콘 플레이북 + 실행 속도 |
| 전략 수립 | 경쟁, 성장, 확장 전략 | 핵심 엔진 + 인지 무기고 + 안티프래질 |
| 문제 해결 | 복잡한 문제 분해 및 해결 | 핵심 엔진 + 인지 무기고 |
| 의사 결정 | 중요한 선택지 분석 | 인지 무기고 + 실행 속도 + 안티프래질 |
| 기회 분석 | 시장/기술 기회 평가 | 핵심 엔진 + 패턴 합성 + 유니콘 플레이북 |
| 체계적 발명 | 기존 시스템/제품의 혁신적 개선 | TRIZ + 핵심 엔진 |

- 핵심 엔진 모듈 선택 시, 10개 엔진 중 **가장 적합한 2~3개**를 선택하라:

| # | 엔진명 | 적합한 상황 |
|---|--------|------------|
| 1 | 제1원리 분해 | 고정관념 탈피, 비용 구조 혁신, 기술 타당성 평가 |
| 2 | 비대칭 기회 탐지 | 100x 수익 기회 발굴, 리스크/보상 비대칭 |
| 3 | 네트워크 효과 설계 | 플랫폼 비즈니스, 사용자 증가→가치 증가 구조 |
| 4 | 시장 창조 | 블루오션, 새 카테고리 정의, 미충족 수요 |
| 5 | 해자 구축 | 경쟁 방어, 장기 우위, 진입장벽 |
| 6 | 반직관 검증 | 대중과 다른 관점 검증, 컨센서스 역행 |
| 7 | 가치 포착 설계 | 비즈니스 모델, 수익 모델, 가격 전략 |
| 8 | 타이밍 인텔리전스 | "왜 지금인가?" 분석, 시장 진입 타이밍 |
| 9 | 복합 우위 적층 | 작은 우위의 복리 효과, 플라이휠 설계 |
| 10 | 생태계 설계 | 플랫폼→생태계 전환, 파트너 네트워크 |

### 1단계 보충: 다양성 인식

모듈 선택 전, 진화 상태의 "모듈 다양성 지표"를 확인하라:
1. H 스코어와 인사이트 원천 분포를 읽는다.
2. 가장 과대표된 모듈을 **제외한다면**, 어떤 모듈이 가장 효과적인 대안인지 검토한다.
3. 위 검토를 참고하여 모듈을 선택한다. 최종 선택은 주제 적합성이 최우선이다.

### 1.5단계: 데이터 주입 판단 (기본 Deep 모드 전용)

> `search` 모드에서는 이미 라우팅 단계에서 검색이 완료되었으므로 이 단계를 건너뛴다.

주제 유형에 따라 WebSearch 자동 호출 여부를 판단하라:

| 주제 유형 | 검색 필요 | 이유 |
|----------|----------|------|
| 아이디어 발굴 | **예** — 자동 호출 | 시장 데이터, 기존 사례가 아이디어 품질을 높임 |
| 사업 구축 | **예** — 자동 호출 | TAM, 경쟁 현황, 시장 검증 데이터 필요 |
| 전략 수립 | **예** — 자동 호출 | 경쟁사, 시장 동향 데이터 없이 전략은 공허함 |
| 기회 분석 | **예** — 자동 호출 | 시장 규모, 타이밍 데이터가 핵심 |
| 문제 해결 | **아니오** — 스킵 | 프레임워크 기반 구조적 분석이 핵심 |
| 의사 결정 | **조건부** — 외부 사실 의존 시 호출 | 멘탈 모델이 핵심이나, 외부 사실에 의존하는 의사결정은 검색 필요 |
| 체계적 발명 | **아니오** — 스킵 | TRIZ 모순 해결은 내부 논리로 충분 |

**자동 호출 실행**: `WebSearch("<핵심 키워드 조합>")`
- WebSearch 도구가 사용 불가한 환경이면 폴백으로 직행한다 (검색 없이 내부 모듈만 사용)
- **폴백**: 실패 또는 도구 미사용 시 SEARCH_DATA = "검색 실패", SEARCH_MODE = "검색 실패 — 내부 모듈만 사용"
- **성공 시**: SEARCH_DATA = 핵심 3~5개 포인트로 압축 (총 500자 이내), SEARCH_MODE = "search 모드 — WebSearch 사용"
- **검색 절약**: 세션당 **1회**로 제한. 추가 필요 시 직접 WebSearch를 사용하세요.

**스킵 시**: SEARCH_DATA = "없음", SEARCH_MODE = "내부 모듈만 사용"

### 2단계: 서브에이전트 스폰

1. 사전 단계에서 이미 읽은 `@references/thinker-prompt.md` 내용을 사용한다
2. 치환 변수를 채운다:
   - `{TOPIC}` → 주제
   - `{CYNEFIN}` → 0단계 Cynefin 진단 결과
   - `{CLASSIFICATION}` → 1단계 분류 유형
   - `{SELECTED_MODULES}` → 선택한 모듈 이름 목록
   - `{SELECTED_ENGINES}` → 선택한 엔진 이름 목록 (핵심 엔진 미선택 시 "해당없음")
   - `{SEARCH_DATA}` → 검색 결과 또는 "없음"
   - `{SEARCH_MODE}` → 검색 모드 상태 문자열
   - `{EVOLUTION_STATE}` → 진화 상태 파일 내용 또는 "없음"
   - `{REFERENCES_DIR}` → 사전 단계에서 해석한 references 절대경로
   - `{DATA_DIR}` → 사전 단계에서 해석한 .data 절대경로
3. 서브에이전트를 스폰한다:
   ```
   Task(
     subagent_type: "general-purpose",
     max_turns: 30,
     prompt: <치환 완료된 프롬프트>
   )
   ```
   > 참고: model 파라미터를 생략하면 현재 세션의 모델이 사용된다. Opus 이상 권장.

### 3단계: 결과 표시

- 서브에이전트의 출력을 **그대로** 사용자에게 표시한다
- 서브에이전트가 brief.md를 생성했으면:
  1. brief.md를 Read로 읽어 `## 실행 대상` 섹션 존재 여부를 확인한다
  2. `## 실행 대상` 섹션이 **있으면** (구현 가능한 작업 포함):
     ```
     ---
     Brief 저장: `{DATA_DIR}/briefs/{YYYY-MM}/{dir-name}/brief.md`
     실행 대상이 감지되었습니다. 실행 계획을 직접 작성하시겠습니까?
     ---
     ```
  3. `## 실행 대상` 섹션이 **없으면** (분석 전용 brief):
     ```
     ---
     Brief 저장: `{DATA_DIR}/briefs/{YYYY-MM}/{dir-name}/brief.md`
     분석 전용 brief입니다.
     ---
     ```

## Light 모드 실행

Light 모드는 레퍼런스를 로딩하지 않고 인컨텍스트에서 경량 분석을 수행한다. 메인 컨텍스트 ~5% 소비.

### 실행 절차

1. **진화 상태 로딩**: `{DATA_DIR}/evolution-state.md`를 읽는다 (존재 시). `## 핵심 인사이트` 헤더가 없으면 무시한다.
2. **Cynefin 진단**: 주제를 Cynefin으로 분류한다
   - **명확(Clear)**: 간단히 답변하고 종료
3. **경량 분석**: 레퍼런스를 읽지 않고 아래만 사용한다:
   - **제1원리 분해**: 주제의 기저 진실을 추출하고 재조합
   - **Cynefin 기반 라우팅**: 도메인에 맞는 사고 방향 설정
   - **진화 상태의 과거 인사이트**: 관련 있는 인사이트 활용
4. **아이디어 생성**: 3~5개 핵심 아이디어 (Deep 대비 축소)
5. **출력**: 아래 경량 형식으로 표시

### Light 출력 형식

```markdown
## SmartThink Light 분석: [주제]

### Cynefin 진단: [도메인]
### 분석 (제1원리 기반)
(핵심 분석 — 레퍼런스 미사용)

### 아이디어 (3~5개)
1. **[제목]**: [설명 2문장] — 실행 가능성: [상/중/하]
2. ...

### Next Steps
1. ...

> Deep 분석이 필요하면 `/smartthink [주제]` 를 실행하세요.
```

### Light 모드에서 하지 않는 것
- 레퍼런스 모듈 Read (핵심 엔진, TRIZ, 인지 무기고 등)
- WebSearch 호출
- 10개 이상 아이디어 생성
- 메타인지 기록 및 진화 상태 갱신
- Brief 내보내기

## 가이드라인

- **서브에이전트 격리**: Deep 모드에서 전체 분석은 서브에이전트의 독립 컨텍스트에서 수행된다. 메인 에이전트는 진단+모듈 선택+스폰+결과 표시만 한다.
- **max_turns**: 서브에이전트 max_turns는 30. 이 값은 9개 레퍼런스 중 2~3개 Read + 전체 분석 + 진화 갱신에 충분한 여유를 포함한다.
- **Cynefin 필수**: Deep/Light 모두 Cynefin 진단이 첫 단계. 명확(Clear) 도메인이면 SmartThink를 건너뛴다.
- **검색 통합**: 검색 데이터는 사고 프레임워크의 **현실 검증 수단**이다. 프레임워크의 논리적 흐름이 항상 우선한다.
- **Light 품질 하한**: Light 모드라도 Cynefin 진단 + 제1원리 분해는 보장한다. 피상적 답변 금지.
- **Brief 내보내기**: 서브에이전트가 구현 가능한 주제 분석 후 brief.md를 생성한다.
- **진화 상태**: 서브에이전트가 Deep 분석 후 반드시 `{DATA_DIR}/evolution-state.md`를 갱신한다. Light 모드에서는 갱신하지 않는다.

# 실행 속도 (Execution Velocity)

> 통찰을 현실로 전환하는 최대 속도 시스템.
> 스타트업에서 속도는 전략적 선택이 아니라 생존 조건이다.
> "First mover advantage"는 신화지만, "Fastest learner advantage"는 물리 법칙이다.

---

## 1. OODA 루프 최적화 (OODA Loop Optimization)

### 개요
John Boyd의 OODA 루프(Observe-Orient-Decide-Act)는 전투기 조종사를 위해 만들어졌지만,
비즈니스에서 가장 강력한 경쟁 프레임워크다.
**OODA 루프가 경쟁자보다 빠르면, 경쟁자는 당신의 2턴 전 행동에 대응하게 된다.**

### OODA 4단계 실행법

#### Observe (관찰): 신호 수집 시간 최소화
- **실시간 데이터 파이프라인**: 고객 행동 → 대시보드까지 지연 시간 < 1시간
- **고객 피드백 루프**: NPS/CSAT를 주간 수집, 월간 아닌 주간
- **경쟁 모니터링**: 주요 경쟁자 3사의 제품 변경/가격 변경/채용 공고를 실시간 추적
- **최적화 방법**: 수동 리포트 → 자동 대시보드 → 실시간 알림 순으로 진화
- **목표**: 관찰에서 인지까지 24시간 이내

#### Orient (정향): 해석 품질 극대화
- 가장 중요하고 가장 느린 단계 — 여기서 인지 편향이 발생한다
- **레드팀 사고**: 모든 중요 해석에 대해 "왜 이것이 틀릴 수 있는가?"를 질문
- **프레임워크 적용**: 데이터를 raw로 보지 않고, 가설-검증 프레임으로 해석
- **다양한 관점**: 최소 3개 부서/역할의 해석을 수집 (엔지니어 + PM + 디자이너)
- **최적화 방법**: 주간 "해석 세션" (30분) — 데이터 → 가설 → 실험 설계
- **목표**: 48시간 내에 데이터를 실행 가능한 가설로 전환

#### Decide (결정): 의사결정 지연 제거
- 의사결정 지연(Decision Delay)이 가장 큰 속도 병목
- **Bezos의 70% 규칙**: 정보 70%에서 결정. 90% 기다리면 너무 느림.
- **결정 소유자 명확화**: 모든 결정에 DRI(Directly Responsible Individual) 1명 지정
- **결정 기한 강제**: 모든 결정에 데드라인 부여. 데드라인 없는 결정은 결정이 아니다.
- **최적화 방법**: "이 결정을 1시간 안에 해야 한다면?" 사고 실험 활용
- **목표**: Type 2 결정(가역적)은 당일, Type 1 결정(비가역적)은 1주일 이내

#### Act (실행): 실행에서 완료까지 마찰 제거
- **배포 파이프라인 자동화**: 코드 머지 → 프로덕션까지 < 30분
- **실험 인프라**: 새 기능 → A/B 테스트까지 < 1일
- **기능 플래그(Feature Flag)**: 배포와 출시를 분리 → 언제든 롤백 가능
- **최적화 방법**: 실행 단계의 모든 수동 단계를 목록화하고 하나씩 자동화
- **목표**: 결정에서 시장 반응 확인까지 1주일 이내

### OODA 속도 벤치마크
| 회사 | OODA 주기 | 비결 |
|------|----------|------|
| Amazon | 하루 수천 건 결정 | Two-way door 결정은 팀 레벨에서 즉시 실행 |
| Meta | 1주일 실험 주기 | 모든 기능을 A/B 테스트, 데이터 기반 결정 |
| ByteDance | 며칠 단위 콘텐츠 실험 | 알고리즘 + A/B 테스트의 극한 조합 |
| Toyota | 분 단위 생산 라인 조정 | Andon cord: 누구든 문제 발견 시 즉시 라인 정지 |

### OODA 자가 진단

```
■ OODA 루프 속도 진단
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
관찰 → 인지: ___시간 (목표: < 24h)
인지 → 가설: ___시간 (목표: < 48h)
가설 → 결정: ___시간 (목표: < 24h / Type 2)
결정 → 실행: ___시간 (목표: < 168h / 1주)
실행 → 결과확인: ___시간 (목표: < 168h / 1주)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
총 OODA 주기: ___일 (목표: ≤ 14일)
병목 구간: [관찰/정향/결정/실행]
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

## 2. 의사결정 프레임워크 (Decision Framework)

### 핵심 원칙
모든 결정이 같은 비중은 아니다. 결정의 유형을 먼저 분류하고, 유형에 맞는 속도로 결정한다.

### Bezos의 문 분류법 (Door Classification)

#### Type 1: 편도 문 (One-Way Door) — 비가역적 결정
- **특성**: 되돌리기 어렵거나 불가능. 비용이 크다.
- **예시**: 핵심 기술 스택 변경, 대규모 펀드레이징, 핵심 인재 해고, 시장 피벗
- **실행법**:
  1. 충분한 데이터 수집 (최소 80%)
  2. 레드팀 + 프리모텀(Pre-mortem) 실행: "이 결정이 대실패한다면 원인은?"
  3. 이해관계자 합의 구축
  4. 결정 + 근거 문서화 (ADR: Architecture Decision Record)
  5. **시간 제한**: 최대 1~2주. 완벽한 정보는 오지 않는다.

#### Type 2: 양방향 문 (Two-Way Door) — 가역적 결정
- **특성**: 되돌릴 수 있다. 실험 가능하다.
- **예시**: UI 변경, 가격 테스트, 마케팅 채널 시도, 새 기능 프로토타입
- **실행법**:
  1. 70% 정보에서 즉시 결정
  2. DRI 1명이 단독 결정 (합의 불필요)
  3. 실행 → 측정 → 조정 (또는 롤백)
  4. **시간 제한**: 당일 또는 다음 영업일 이내.

### 의사결정 속도 공식

```
DV (Decision Velocity) = (Q × S) / D

Q = Quality (결정 품질, 0-1)
S = Speed (결정 속도, decisions/week)
D = Cost_of_delay (지연 비용, $/day)
```

- 대부분의 조직이 Q를 0.95→0.99로 올리기 위해 S를 절반으로 줄인다.
- 하지만 Q가 0.7이어도 S가 5배 빠르면 DV가 3.5배 높다.
- **교훈**: 빠르게 결정하고 빠르게 고치는 것이, 느리게 결정하고 맞추는 것보다 낫다.

### 70% 정보 규칙 실천법

1. **결정에 필요한 정보 목록** 작성 (5~10개)
2. **현재 보유한 정보**에 체크
3. **70% 이상** 체크되면 → 결정 실행
4. **70% 미만**이면 → 가장 빠르게 얻을 수 있는 정보 1~2개만 추가 수집
5. 수집 후에도 100%에 도달하지 않을 것 — 이것이 정상이다

### 의사결정 안티패턴

| 안티패턴 | 증상 | 해결 |
|----------|------|------|
| 분석 마비 (Analysis Paralysis) | "데이터가 더 필요해" 반복 | 70% 규칙 강제 + 결정 데드라인 |
| HiPPO (Highest Paid Person's Opinion) | 직급이 결정 | DRI 제도 + 데이터 기반 문화 |
| 합의 중독 (Consensus Addiction) | 모든 사람이 동의해야 진행 | RACI 매트릭스: Approve 1명만 |
| 결정 회피 (Decision Avoidance) | "다음 주에 논의하자" | 비결정의 비용(Cost of Indecision) 명시 |

## 3. 최소 실행 가능 단위 (Minimum Viable Everything)

### 핵심 원칙
MVP는 제품에만 적용되는 것이 아니다. 모든 활동에 "최소 실행 가능 단위"를 적용한다.
가장 큰 가정(Riskiest Assumption)을 가장 작은 실험(Smallest Test)으로 검증한다.

### MVx 프레임워크

#### MVT: 최소 실행 가능 테스트 (Minimum Viable Test)
- 모든 가설에 대해 가장 빠르고 저렴한 테스트를 설계
- **구조**: 가설 → 지표 → 테스트 방법 → 성공/실패 기준 → 소요 시간
- **예시**:
  ```
  가설: 프리랜서가 세금 계산에 월 5시간 이상 쓰고 있다
  지표: 프리랜서 100명 중 세금 도구에 $30/월 이상 지불 의향
  테스트: 구글 폼 설문 + 타겟 커뮤니티 공유
  성공 기준: 응답자 40%+ "반드시 사용" 또는 "매우 관심"
  소요 시간: 3일
  ```

#### MVP 유형별 실행 가이드

| 유형 | 소요 시간 | 비용 | 검증 수준 | 최적 상황 |
|------|----------|------|----------|-----------|
| **Smoke Test** (랜딩페이지) | 1~3일 | < $500 | 수요 | 시장이 존재하는지 모를 때 |
| **Wizard of Oz** | 1~2주 | < $2K | 가치 | 자동화 전에 프로세스 검증 |
| **Concierge** | 2~4주 | 인건비 | 가치 + 과정 | 최적 프로세스를 모를 때 |
| **기능적 MVP** | 4~8주 | $5K-$50K | 전체 | PMF 검증 준비 완료 시 |

#### 가장 큰 가정 우선 (Riskiest Assumption Test)
- 모든 가설을 나열한다
- 각 가설을 (영향도 × 불확실성)으로 점수 매긴다
- 가장 높은 점수의 가설부터 테스트한다
- **이유**: 가장 위험한 가설이 통과하지 못하면 나머지는 의미 없다

```
■ 가설 우선순위 매트릭스
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
| 가설 | 영향도(1-5) | 불확실성(1-5) | 점수 | 순서 |
|------|-----------|-------------|------|------|
|      |           |             |      |      |
|      |           |             |      |      |
|      |           |             |      |      |
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
→ 점수 높은 순서로 테스트 실행
```

### 실전 사례: Zappos
- **가장 큰 가설**: "사람들이 온라인으로 신발을 살 것인가?" (1999년, 무모한 질문)
- **MVT**: Nick Swinmurn이 지역 신발 매장에서 사진을 찍어 웹사이트에 올림
- **프로세스**: 주문이 들어오면 직접 매장에 가서 정가로 구매 후 배송
- **비용**: 웹사이트 개발비 + 시간 (재고 비용 $0)
- **결과**: 사람들이 실제로 구매함 → 가설 검증 → 본격 투자 → Amazon에 $1.2B 인수

## 4. 반복 속도가 곧 해자 (Iteration Speed as Moat)

### 핵심 원칙
경쟁 우위는 제품이 아니라 **학습 속도**에서 나온다.
같은 시간에 더 많이 실험하고 더 빨리 배우는 팀이 결국 이긴다.

### 학습 속도 공식 (Learning Velocity Formula)

```
LV (Learning Velocity) = E × L

E = Experiments/week (주당 실험 횟수)
L = Learning/experiment (실험당 학습량, 0-1)
```

- **E 높이는 방법**: 실험 셋업 시간 단축, 병렬 실험, 실험 인프라 투자
- **L 높이는 방법**: 명확한 가설과 성공 기준 사전 정의, 실험 후 회고 필수

### 배포 빈도 = 경쟁 우위

| 회사 | 배포 빈도 | 방법 |
|------|----------|------|
| Amazon | 11.7초마다 1회 (2015) | 마이크로서비스 + 자동 배포 + 기능 플래그 |
| Netflix | 수천 회/일 | Spinnaker + 카나리 배포 + 자동 롤백 |
| Etsy | 50+ 회/일 | 개발자가 직접 배포 + 모니터링 |
| Facebook | 1,000+ 회/일 | Gatekeeper (기능 플래그) + 점진적 롤아웃 |

- 배포가 빈번하다 = 변경 크기가 작다 = 리스크가 작다 = 롤백이 빠르다
- 연 4회 릴리스 기업이 연 1,000회 릴리스 기업을 이기는 것은 불가능

### A/B 테스트 체계

- **전제**: "의견이 아니라 데이터로 결정한다" (Data > HiPPO)
- **실험 계획 구조**:
  ```
  실험명: [___]
  가설: [X를 하면 Y 지표가 Z% 향상될 것이다]
  주요 지표: [1개만. 보조 지표는 별도.]
  대상: [트래픽의 __% / 특정 세그먼트]
  기간: [통계적 유의성 달성까지 예상 __일]
  성공 기준: [p < 0.05, 효과 크기 > ___%]
  실패 시 계획: [___]
  ```
- **주의사항**:
  - 동시에 너무 많은 실험 = 상호 간섭 → 실험 격리 또는 다변량 테스트 사용
  - 최소 탐지 효과(MDE)를 사전에 계산 → 트래픽 부족하면 정성 테스트로 대체
  - "통계적으로 유의하지 않음" ≠ "효과 없음" → 표본 크기 재계산

### 실험 문화 구축 4단계

1. **인프라 구축**: A/B 테스트 도구 + 기능 플래그 + 자동 분석 대시보드
2. **모든 팀이 실험**: 엔지니어링만이 아니라 마케팅, 영업, CS도 실험
3. **실패 축하**: 실험 실패 = 학습 성공. "이번 분기 최고의 실패" 시상
4. **실험 회고 의무화**: 모든 실험 종료 후 "무엇을 배웠는가?" 1페이지 작성

## 5. 자원 배분 알고리즘 (Resource Allocation Algorithm)

### 핵심 원칙
자원은 항상 부족하다. 모든 "예"는 다른 모든 것에 대한 "아니오"다.
배분의 핵심은 "무엇을 할 것인가"가 아니라 "무엇을 하지 않을 것인가"이다.

### 70-20-10 규칙 (Bezos/Google Allocation)

| 영역 | 비율 | 성격 | 예시 |
|------|------|------|------|
| **핵심 (Core)** | 70% | 현재 수익원 강화 | 기존 제품 개선, 핵심 기능 최적화 |
| **인접 (Adjacent)** | 20% | 기존 역량의 새 적용 | 새 시장 진출, 인접 제품 |
| **혁신 (Moonshot)** | 10% | 높은 불확실성, 높은 임팩트 | 신기술 실험, 시장 창출 |

- 70%를 소홀히 하면 생존이 위험하고, 10%를 안 하면 미래가 없다
- 핵심 비즈니스의 성장률이 둔화될수록 20%와 10%의 비중을 높여라

### OKR 기반 우선순위 결정

**분기별 프로세스**:
1. **회사 OKR** 설정 (Objective 3개 이하, KR 각 3개 이하)
2. **팀 OKR** 정렬 (회사 OKR에서 cascading)
3. **모든 프로젝트/태스크**를 OKR에 매핑
4. **OKR에 매핑되지 않는 것** → 하지 않는다 (가장 중요한 단계)
5. **주간 체크인**: KR 진행률 리뷰, 장애물 제거

### "Hell Yes or No" 프레임워크 (Derek Sivers)

- 새로운 기회, 요청, 프로젝트에 대한 반응이 "Hell Yes!"가 아니면 → "No"
- 적용 기준:
  - "이것이 핵심 지표를 2배 이상 움직이는가?" → Hell Yes가 아니면 No
  - "이것을 안 하면 6개월 후 후회하는가?" → 아니면 No
  - "이것이 현재 가장 큰 병목을 해결하는가?" → 아니면 No

### 기회비용 사고 (Opportunity Cost Thinking)

- 모든 시간/돈/인력 투자에 대해 물어본다: "같은 자원으로 할 수 있는 최선의 대안은?"
- **시간 기회비용**: 엔지니어 1명 × 1주일 = 가장 높은 ROI 프로젝트에 투입되어야 한다
- **주의력 기회비용**: CEO/리더의 관심은 가장 희소한 자원. 매일 "오늘 내가 집중할 1가지"를 결정.

### 킬 기준 (Kill Criteria)

프로젝트 시작 전에 "언제 멈출 것인가"를 미리 정한다:

```
■ 프로젝트 킬 기준
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
프로젝트: [___]
시작일: [___]
검증 기한: [___]

킬 조건 (하나라도 해당 시 중단 검토):
  [ ] ___ 기한까지 ___지표가 ___에 미달
  [ ] 누적 투자가 $___ 초과 시 ROI 재평가
  [ ] 핵심 가설 [___]이 부정될 경우

킬 결정 프로세스:
  1. 킬 조건 해당 확인
  2. 피벗 가능성 검토 (24시간 내)
  3. 피벗 불가 시 자원 재배분
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

### 실전 사례: Apple의 자원 배분
- Steve Jobs 복귀(1997) 후 첫 행동: 제품 라인 350개 → 10개로 축소
- "집중(Focus)이란 집중할 것을 고르는 것이 아니라, 나머지 100가지에 'No'라고 말하는 것이다"
- 결과: 거의 파산 → 20년 만에 세계 최고 시가총액 기업 ($3T+)

## 6. 속도의 복리 (Compounding Speed)

### 핵심 원칙
속도는 복리(Compound)로 작동한다. 매주 1%씩 빨라지면, 1년 후 67% 빨라진다.
빠른 반복 → 더 많은 학습 → 더 나은 결정 → 더 빠른 반복 → ... (자기 강화 루프)

### 속도 복리 루프

```
┌─────────────────────────────────────────┐
│                                         │
│  빠른 반복 (Fast Iterations)            │
│       ↓                                 │
│  더 많은 실험 (More Experiments)         │
│       ↓                                 │
│  더 빠른 학습 (Faster Learning)          │
│       ↓                                 │
│  더 나은 결정 (Better Decisions)         │
│       ↓                                 │
│  더 효율적인 실행 (More Efficient Execution) │
│       ↓                                 │
│  더 빠른 반복 ... (repeat)              │
│                                         │
└─────────────────────────────────────────┘
```

### 조직 속도 구축 7가지 레버

#### 레버 1: 승인 레이어 축소 (Reduce Approval Layers)
- **문제**: 결정이 3계층 승인을 거치면, 각 계층에서 2일 지연 = 6일 낭비
- **해결**: Amazon "Two-Pizza Team" — 팀이 자율적으로 결정. 보고하지 말고 실행하라.
- **실행**: 결정 권한 매트릭스 작성 → $X 이하 / 리스크 등급 Y 이하는 팀 레벨 자율 결정

#### 레버 2: 팀 자율성 (Team Autonomy)
- **문제**: 모든 결정을 CEO/CTO가 내리면 병목 발생
- **해결**: "Context, not control" (Netflix) — 리더는 컨텍스트를 주고, 결정은 팀이 한다
- **실행**: OKR로 방향 설정 → 팀이 How를 결정 → 결과로 평가

#### 레버 3: 피드백 루프 단축 (Short Feedback Loops)
- **문제**: 분기별 리뷰 = 90일의 블라인드 실행
- **해결**: 주간 스프린트 + 일간 스탠드업 + 실시간 지표 대시보드
- **실행**: 핵심 지표 3개를 실시간 대시보드에 표시. 매일 15분 체크.

#### 레버 4: 속도 도구 투자 (Velocity Tooling)
- **문제**: 개발자가 빌드 대기에 하루 30분, 배포에 1시간, 환경 셋업에 반나절
- **해결**: CI/CD 파이프라인, 개발 환경 자동화, 내부 도구 투자
- **실행**: "개발자 경험(Developer Experience)" 전담 → 마찰 요소 top 5 제거
- **ROI**: 엔지니어 50명 × 하루 30분 절약 = 연간 엔지니어 3명분의 시간

#### 레버 5: 비동기 우선 커뮤니케이션 (Async-First Communication)
- **문제**: 미팅이 하루 4시간 = 생산적 시간 50% 감소
- **해결**: 기본은 비동기(문서, 메시지). 동기(미팅)는 예외적으로만.
- **실행**: "이 미팅은 문서로 대체 가능한가?" → 가능하면 문서로 전환

#### 레버 6: 기술 부채 관리 전략 (Technical Debt Strategy)
- **초기 (Pre-PMF)**: 기술 부채를 의도적으로 허용한다. 속도 > 코드 품질.
- **PMF 후**: 스프린트의 20%를 기술 부채 상환에 할당.
- **스케일링**: 기술 부채가 배포 속도를 늦추기 시작하면 즉시 인프라 투자.
- **기준**: "이 기술 부채가 배포 빈도를 줄이는가?" → Yes이면 즉시 해결.

#### 레버 7: 의도적 단순화 (Intentional Simplification)
- 복잡성은 속도의 적이다. 시스템, 프로세스, 조직 모두에 적용.
- **코드**: "이 코드를 새 팀원이 30분 안에 이해할 수 있는가?" → No이면 리팩토링
- **프로세스**: "이 프로세스에서 단계를 하나 제거할 수 있는가?" → 분기별 프로세스 감사
- **조직**: "이 팀/역할이 속도를 높이는가, 늦추는가?" → 냉정하게 평가

### 속도 vs 품질: 가짜 트레이드오프

- **초기 단계 (0→1)**: 속도와 품질은 트레이드오프가 아니다. 둘 다 할 수 있다.
  - 작은 범위(Scope) → 빠르게 만들 수 있고 + 품질도 높을 수 있다
  - 핵심은 범위를 줄이는 것이지, 품질을 줄이는 것이 아니다
- **스케일링 단계 (1→N)**: 속도를 위해 품질을 희생하면 기술 부채로 돌아온다
  - "Move fast with stable infrastructure" (Facebook 2014 이후 모토 변경)
  - 속도를 유지하면서 품질을 올리는 유일한 방법: 자동화 (테스트, CI/CD, 모니터링)

## 7. 블리츠스케일링 조건 판단 (Blitzscaling Decision Matrix)

### 개요
Reid Hoffman의 블리츠스케일링(Blitzscaling): 불확실성 속에서 효율보다 속도를 극단적으로 우선하는 전략.
모든 기업에 적용되는 것이 아니다. 잘못 적용하면 가장 빠르게 망하는 방법이다.

### 블리츠스케일링 적합성 진단

#### 필수 조건 (3개 모두 충족 시에만 실행)

| 조건 | 질문 | Yes 기준 |
|------|------|----------|
| **승자독식 시장** | 1등이 시장의 60%+ 가져가는가? | 네트워크 효과 + 규모의 경제 존재 |
| **선점 우위** | 먼저 규모를 달성하면 후발주자가 따라잡기 어려운가? | 전환 비용 + 네트워크 효과 강함 |
| **현재 위치** | 1~2등 안에 들 현실적 가능성이 있는가? | 자금 + 팀 + 트랙션 기준 |

#### 부적합 조건 (하나라도 해당 시 블리츠스케일링 금지)

- 시장이 분산적 (취향/지역 기반, 다수의 플레이어 공존)
- 제품이 범용재 (Commodity) — 가격 경쟁만 남음
- 네트워크 효과 없음 — 규모가 커져도 사용자 가치 불변
- PMF 미달성 — 블리츠스케일링 ≠ PMF 달성 전략

### 번레이트 관리 (Burn Rate Management)

#### 안전한 번(Burn) 수준 계산

```
■ 번레이트 관리 프레임워크
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
현재 현금: $___
월간 번레이트: $___
런웨이: ___ 개월

안전 규칙:
  런웨이 ≥ 18개월 → 정상 운영
  12-18개월 → 주의 (효율화 또는 펀드레이징 시작)
  6-12개월 → 경보 (즉시 펀드레이징 + 비용 절감)
  < 6개월 → 위기 (생존 모드: 핵심 팀만 유지)

블리츠스케일링 번레이트 기준:
  월간 매출 성장률 > 월간 번레이트 성장률 → 허용
  반대 → 위험 (유닛 이코노믹스 재점검)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

### "Default Alive" 테스트 (Paul Graham)

- 질문: "현재 성장률과 현재 비용이 유지된다면, 투자 없이 흑자 전환이 가능한가?"
- **Default Alive** → 안전한 위치. 블리츠스케일링 선택 가능.
- **Default Dead** → 위험한 위치. 블리츠스케일링은 도박.
- 계산법:
  ```
  월간 수익 성장률: g%
  월간 비용: C
  현재 월간 수익: R
  도달 월수: R × (1+g)^n = C → n = log(C/R) / log(1+g)
  현재 런웨이 내에 n에 도달하면 → Default Alive
  ```

### 블리츠스케일링 단계별 실행 원칙

| 단계 | 규모 | 핵심 원칙 | 수용하는 비효율 |
|------|------|----------|----------------|
| **Family** | 1~9명 | "Do things that don't scale" | 모든 것을 수동으로 |
| **Tribe** | 10~99명 | "Build the machine that builds the machine" | 불완전한 프로세스 |
| **Village** | 100~999명 | "Speed > Efficiency, but build guardrails" | 인력 과잉 고용 |
| **City** | 1,000~9,999명 | "Manage controlled chaos" | 조직 중복, 내부 경쟁 |
| **Nation** | 10,000명+ | "Maintain founder's mentality" | 관료주의와의 전쟁 |

### 블리츠스케일링 실전 사례: Uber

- **시장 조건**: 승자독식(네트워크 효과: 운전자↔승객) + 선점 우위(도시별 규모 달성 시 방어 가능)
- **실행**: 도시 하나씩 블리츠스케일 → 운전자 보조금 + 승객 할인 = 대규모 적자
- **번레이트**: 최대 월 $1B 이상 손실 (의도적)
- **근거**: 각 도시에서 규모 달성 → 네트워크 효과 → 경쟁자 진입 방어 → 장기 수익성
- **결과**: 75개국 진출, $70B+ IPO (2019). 다만, 수익성 달성까지 10년+ 소요.
- **교훈**: 블리츠스케일링은 "빠르게 쓰는 것"이 아니라 "선점으로 방어 가능한 위치에 빠르게 도달하는 것"이다.

### 블리츠스케일링 의사결정 체크리스트

```
■ 블리츠스케일링 Go/No-Go 체크리스트
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
필수 조건 (모두 Yes 필수):
  [ ] 승자독식 시장인가?
  [ ] 선점 우위가 구조적으로 존재하는가?
  [ ] PMF가 달성되었는가?
  [ ] 충분한 자금이 있거나 조달 가능한가?
  [ ] Default Alive 이거나 충분한 런웨이(18개월+)가 있는가?

부적합 신호 (하나라도 Yes면 중단):
  [ ] 시장이 분산적/지역적인가?
  [ ] 네트워크 효과가 없는가?
  [ ] 경쟁이 가격 경쟁으로만 이루어지는가?
  [ ] 유닛 이코노믹스가 규모에서도 개선되지 않는가?

판정:
  필수 조건 5/5 + 부적합 0/4 → GO
  그 외 → NO (효율적 성장 전략으로 전환)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```
